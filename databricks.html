Complete Guide to Databricks and Delta Lake

This document explains the core concepts behind Databricks and the Lakehouse architecture. The goal is to help a reader understand how data flows from raw files to analytics-ready datasets.

1. What Databricks Actually Is

Databricks is a cloud platform built on Apache Spark that helps engineers process, manage, and analyze large amounts of data.

However, Databricks itself does not store your data. The data lives in cloud storage systems such as S3, ADLS, or Google Cloud Storage. Databricks provides the compute engine and management layer.

Basic Architecture

+--------------------+ | BI Tools | | (PowerBI/Tableau) | +----------+---------+ | SQL Warehouse | +---------+---------+ | Databricks | | Spark Clusters | +---------+---------+ | Unity Catalog | +---------+---------+ | Cloud Storage | | S3 / ADLS / GCS | +-------------------+

Think of Databricks as the compute engine that reads and writes data stored in the cloud.

2. Where the Data Is Actually Stored

The physical data files are stored in cloud object storage.

Cloud Provider Storage System Example Path
AWS S3 s3://company-datalake/orders/
Azure ADLS Gen2 abfss://data@company.dfs.core.windows.net/orders/
Google Cloud GCS gs://company-datalake/orders/

These locations store the actual files such as Parquet data and Delta transaction logs.

3. Unity Catalog (Data Governance Layer)

Unity Catalog is the centralized governance system used by Databricks. It organizes and secures all data assets.

Unity Catalog | +-- Catalog | +-- Schema (Database) | +-- Tables +-- Views +-- Functions +-- Volumes

Example

analytics.sales.orders

This means:

  • Catalog: analytics
  • Schema: sales
  • Table: orders

Unity Catalog allows fine-grained access control.

GRANT SELECT ON TABLE analytics.sales.orders TO analysts;

4. Tables in Databricks

Tables represent structured datasets that can be queried using SQL.

Two types exist:

  • Managed Tables
  • External Tables

5. Managed Tables

Managed tables are fully controlled by Databricks. 

CREATE TABLE sales.orders (
order_id INT,
amount DOUBLE
);

Databricks automatically decides where the data files will be stored.

Behavior

  • Storage location managed automatically
  • Dropping the table deletes the data
  • Best for internal pipelines

6. External Tables

External tables reference data stored in a location you specify. 

CREATE TABLE sales.orders
USING DELTA
LOCATION 's3://company-data/orders/';

Behavior

  • Data location is user controlled
  • Dropping the table does not delete files
  • Common for shared data lakes

7. What Delta Lake Is

Delta Lake is the default storage format used by Databricks.

It builds on Parquet but adds reliability features such as transactions, schema enforcement, and time travel.

Delta Table Folder Structure

orders/ part-0001.parquet part-0002.parquet part-0003.parquet _delta_log/ 000000000000.json 000000000001.json 000000000002.json

Parquet files store the data while the transaction log tracks changes.

8. Delta Lake Transaction Log

The _delta_log folder contains metadata about every operation.

File Meaning
000000000000.json Table creation
000000000001.json Data inserted
000000000002.json Update operation

This enables features such as ACID transactions and version history.

9. Time Travel

Delta Lake allows querying previous versions of a table. 

SELECT *
FROM sales.orders VERSION AS OF 3;

This is extremely useful for debugging data pipelines.

10. Data Lakehouse Medallion Architecture

Databricks pipelines usually follow a layered structure called the Medallion Architecture.

Raw Data | v Bronze Layer | v Silver Layer | v Gold Layer

11. Bronze Layer (Raw Data)

Bronze tables store raw data exactly as it arrives.

Examples:
  • API responses
  • CSV files
  • Kafka streams
  • logs
Example table: 
bronze.transactions_raw
Characteristics:
  • Minimal transformation
  • Append-only ingestion
  • Used for auditing

12. Silver Layer (Clean Data)

Silver tables contain cleaned and standardized data.

Typical processing includes:
  • Removing duplicates
  • Fixing schemas
  • Filtering bad records
  • Joining datasets
Example: 
silver.transactions_clean

13. Gold Layer (Business Data)

Gold tables are optimized for analytics and reporting.

Examples:
  • Daily revenue
  • Customer metrics
  • Aggregated KPIs
Example: 
gold.daily_sales
These tables feed dashboards and BI tools.

14. Example Data Pipeline

CSV files in S3 | v Bronze Table (raw ingestion) | v Silver Table (cleaned data) | v Gold Table (aggregated metrics) | v BI Dashboard

15. Putting Everything Together

Cloud Storage (S3 / ADLS / GCS) | +-- Delta Tables | | | +-- Parquet Data Files | +-- _delta_log | +-- Raw Files | +-- CSV +-- JSON +-- Images Unity Catalog | +-- Catalog | +-- Schema | +-- Tables +-- Views +-- Volumes Databricks Compute | +-- Spark Clusters +-- SQL Warehouse

16. Key Takeaways

  • Databricks is a compute platform built on Spark.
  • Actual data lives in cloud storage systems.
  • Unity Catalog manages metadata and permissions.
  • Delta Lake provides reliable tables with transaction logs.
  • Medallion architecture organizes data into Bronze, Silver, and Gold layers.

This document can serve as a reference for engineers learning how modern data platforms work using the Lakehouse architecture.
