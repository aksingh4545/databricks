1. Overview

This project demonstrates a complete end-to-end data pipeline on Azure for real-time streaming processing of movie metrics.

Key Features:
  • Ingestion: Azure Event Hub captures events in Avro format to ADLS
  • Processing: Databricks Structured Streaming with deduplication and windowed aggregations
  • Storage: Delta Lake for ACID-compliant, versioned storage
  • Serving: Cosmos DB for low-latency querying
  • Exactly-Once Semantics: Watermarking, checkpointing, and deduplication

The pipeline processes events with schema including order_id, event_time, imdb_score, aggregating them into 5-minute windows for metrics like average IMDb score and movie count.

3. Prerequisites

  • Azure Subscription with:
    • Event Hub Namespace
    • Azure Data Lake Storage Gen2 account
    • Databricks Workspace
    • Cosmos DB Account (database: analyticsdb, container: movie_metrics)
  • Databricks Runtime: 13.3 LTS or later
  • Libraries: PySpark (built-in), Cosmos DB connector
  • Secrets: Cosmos DB endpoint and key in Databricks Secrets

5. Databricks Queries Handbook

Important: Run queries in order. Batch validation (Query 1) must run before streaming to infer schema.

Query 1: Batch Read for Schema Validation

# Read existing Avro files to infer schema (runs once)
static_df = spark.read \
    .format("avro") \
    .option("recursiveFileLookup", "true") \
    .load(
        "abfss://eventhub-capture@pipelinestoragetask45.dfs.core.windows.net/event-hub45/"
    )

static_df.printSchema()
Why this is required: Structured Streaming does not support schema inference. We must infer schema before starting streaming.

Query 2: Bronze Layer - Streaming Read

# Start continuous streaming job watching ADLS for new files
bronze_df = spark.readStream \
    .format("avro") \
    .schema(static_df.schema) \
    .option("recursiveFileLookup", "true") \
    .load(
        "abfss://eventhub-capture@pipelinestoragetask45.dfs.core.windows.net/event-hub45/"
    )
Key Concept: readStream = "keep listening forever for new files"

Query 3: Understanding Event Hub Capture Payload

Event Hub Capture stores metadata columns plus business payload inside Body (binary):

Bronze Schema:
├── SequenceNumber    (long)
├── Offset            (string)
├── EnqueuedTimeUtc   (timestamp)
└── Body              (binary) ← Business data is HERE

Query 4: Parsing JSON Payload (Silver Foundation)

from pyspark.sql.functions import col, from_json

# Define schema for JSON payload
json_schema = static_df.schema["Body"].dataType

# Parse binary Body to string, then parse JSON
parsed_df = bronze_df.select(
    from_json(col("Body").cast("string"), json_schema).alias("data")
)

silver_df = parsed_df.select("data.*")
Bronze = envelope (metadata + binary Body) Silver = extracted payload (structured columns)

Query 5: Event-Time Preparation

from pyspark.sql.functions import to_timestamp

# Convert event_time from string to timestamp (REQUIRED for watermark)
silver_df = silver_df.withColumn(
    "event_time", to_timestamp(col("event_time"))
)
Critical: Watermark and window functions require timestamp type, not string!

Query 6: Watermark Definition (Late Data Handling)

# Tell Spark to wait 10 minutes for late-arriving data
dedup_df = silver_df \
    .withWatermark("event_time", "10 minutes")
Watermark = Allowed lateness window. Controls how long Spark waits for late data before finalizing results.

Query 7: Deduplication with Watermark

# Remove duplicates using order_id as key
dedup_df = silver_df \
    .withWatermark("event_time", "10 minutes") \
    .dropDuplicates(["order_id"])
Rule: Streaming deduplication MUST be paired with watermark to limit state memory.

Query 8: Window Aggregation (Gold Logic)

from pyspark.sql.functions import window, count

# Group events into 5-minute time windows
agg_df = dedup_df.groupBy(
    window(col("event_time"), "5 minutes")
).count()

Query 9: Advanced Aggregation (Business Metrics)

from pyspark.sql.functions import avg

# Compute business metrics per window
agg_df = dedup_df.groupBy(
    window(col("event_time"), "5 minutes")
).agg(
    avg("imdb_score").alias("avg_imdb_score"),
    count("*").alias("movie_count")
)

Query 10: Writing Streaming Output to Delta Lake

# Start long-running streaming job writing to Delta
query = agg_df.writeStream \
    .format("delta") \
    .outputMode("append") \
    .option(
        "checkpointLocation",
        "abfss://processed@pipelinestoragetask45.dfs.core.windows.net/checkpoints/movie_metrics"
    ) \
    .start(
        "abfss://processed@pipelinestoragetask45.dfs.core.windows.net/gold/movie_metrics"
    )
Key Concept: Delta + append mode + watermark = exactly-once streaming semantics.

Query 11: Checkpoint Mechanics

Checkpoint stores:

  • Processed file offsets
  • Watermark position
  • Window state
  • Deduplication state
Golden Rule: If you change streaming logic, you MUST change the checkpoint path. Old checkpoint will ignore new transformations.

Query 12: Why Delta Table Was Sometimes Empty

If all test events have the same event_time:

  • Spark waits for late data
  • Watermark never advances
  • Window never closes
  • Append mode writes nothing
Empty Delta does NOT mean pipeline failure—it means watermark hasn't advanced enough to close windows.

Query 13: Temporary Validation Using Processing Time

from pyspark.sql.functions import current_timestamp

# TESTING ONLY: Override event_time with current timestamp
silver_test_df = silver_df.withColumn(
    "event_time", current_timestamp()
)
Use Case: This is NOT production logic. Use only to validate pipeline correctness during testing.

Query 14: Reading Gold Delta Table (Batch)

# Batch read of finalized results for BI/Analytics
gold_df = spark.read.format("delta").load(
    "abfss://processed@pipelinestoragetask45.dfs.core.windows.net/gold/movie_metrics"
)

gold_df.display()

Query 15: Writing Gold Data to Cosmos DB

# Batch write to Cosmos DB for low-latency serving
gold_df.write \
    .format("cosmos.oltp") \
    .option("spark.cosmos.accountEndpoint", COSMOS_ENDPOINT) \
    .option("spark.cosmos.accountKey", COSMOS_KEY) \
    .option("spark.cosmos.database", "analyticsdb") \
    .option("spark.cosmos.container", "movie_metrics") \
    .mode("append") \
    .save()

Query 16: Final Mental Model

readStream → infinite input withWatermark → late data control window → bounded aggregation dropDuplicates → exactly-once semantics writeStream → continuous output checkpoint → system memory Delta → safe, queryable storage

8. Production Checklist

Pre-Deployment Checklist:
  1. Environment Setup
    • ✓ Verify ADLS mounts and permissions
    • ✓ Configure secrets for Cosmos DB credentials
  2. Schema Validation
    • ✓ Run Query 1 and confirm schema matches expected fields
  3. Streaming Job Configuration
    • ✓ Use new checkpoint path for each deployment
    • ✓ Set watermark to match expected latency (10 minutes)
    • ✓ Use outputMode("append") for immutable outputs
  4. Monitoring
    • ✓ Enable Databricks alerts for job failures
    • ✓ Monitor watermark progress in Spark UI
    • ✓ Check state size growth over time
  5. Testing
    • ✓ Simulate late data with delayed event_time
    • ✓ Validate empty table scenario (Query 12)
    • ✓ Use processing time override (Query 13) for quick checks
  6. Recovery
    • ✓ Document checkpoint path for each job
    • ✓ Use Delta versioning for rollbacks

9. Troubleshooting

Problem Cause Solution
No data in Delta table Watermark hasn't advanced; windows not closed Ensure event times vary; use Query 13 for testing
Schema mismatch error JSON schema doesn't match actual payload Rerun Query 1 and update json_schema
Duplicates in output Deduplication key not unique Verify order_id is unique; check watermark
Job fails on restart Old checkpoint incompatible with new logic Delete checkpoint or use new path
Late data ignored Watermark delay too short Increase watermark delay (e.g., 10 → 30 minutes)
Cosmos DB write errors Insufficient RUs or wrong credentials Check RU provisioning; verify endpoint/key
For more details: Refer to Databricks Structured Streaming Documentation
